# BayChat AI Reply の根本原則（カテゴリ2レビューで確立・2026-04-29）

## 大原則：80点目標（100点は補足情報があって初めて到達）

補足情報（sellerSetting）なしの AI Reply は **構造的に100点を取れない**。なぜなら、商品ページに記載のない情報（素材・原産地・付属品・状態詳細など）を聞かれた場合、AIに答える根拠がないからである。

- **補足情報なし → 80点が目標**：何が確認できて何ができないかを誠実に言い切る完全で正直な返信
- **補足情報あり → 100点が目標**：セラー意図を反映した完全な返信
- **絶対禁止**：捏造して100点を装うこと（事実の幻覚）

**Why:** 社長指摘 2026-04-29 cat02_02（素材確認）レビュー：「AI Replyは100点は無理！情報がないとわからないから。補足情報があって初めて100点になる。補足情報無しでのAI Replyが目指すのは80点。大前提として記録しておいて」

**How to apply:** プロンプト ROLE に QUALITY GOAL として明文化。FACT GROUNDING で「分からないものは分からない」と素直に言う運用。テスト結果評価時も「捏造していないか」を最重要チェックポイントに。

## 4大ルール（プロンプト v2.3_baseline_c2 に実装済み）

### ルール1：FACT GROUNDING — 事実根拠の厳格化（セラー本人視点）

商品情報・Item specifics・出品画像・補足情報に記載のない事実を **断言してはいけない**。捏造禁止。

**重要：セラー本人として返信している自覚**
出品ページを作ったのはセラー自身。「商品情報に記載がございません」のような **他人事の言い方は禁止**。「私は把握していません」「分かりかねます」と素直に言う。

**OK例**：
- 「製造地については分かりかねます。それ以外は出品ページの内容の通りです」
- 「シリアル番号は手元にありません。教えていただければ製造年・原産地をお調べします」

**NG例**：
- 「商品情報に記載がございません」（他人事）
- 「素材は合成皮革です」（記載なしなのに断言＝幻覚）
- 「元箱と書類が付属しています」（記載なしなのに断言）

**Why:** cat02_02（素材幻覚）・cat02_03（付属品幻覚）・cat02_06（状態幻覚）・cat02_07（原産地幻覚）・cat02_08（オリジナルケース幻覚）で全モデルが事実を捏造していた。

### ルール2：NO USER BURDEN — 追加作業デフォルト禁止

写真追加・採寸など、追加作業をデフォルトで約束しない。バイヤーから要望が来た場合も、補足情報なしでは「対応していません」と断る。

**OK応答**：
「商品ページに掲載している写真以外の追加撮影は行っておりません。商品ページの情報と写真をご確認ください。気になる点があれば具体的に教えていただければできる限りお答えします」

**NG応答**：
- 「追加の写真を撮ってお送りします」（勝手に約束）
- 「採寸して報告します」（勝手に約束）
- 「I'll get back to you」（先送り＋未来約束）

**Why:** cat02_10（追加写真依頼）で 4.1-Mini FRIENDLY「写真撮ります」、5-Mini「全部撮ります、特定箇所を指定してください」など、セラーに追加作業を発生させる約束を勝手に出していた。

### ルール3：NO FALSE POLICY — 誤ポリシー言及禁止

存在しない／該当しない eBay ポリシーを根拠として持ち出さない。実際の運用上の理由（提供していない・サービス外）で断る。

**NG例**：
- 「eBayポリシーにより、追加の写真をお送りすることはできません」（写真追加はポリシー違反ではない）

**OK例**：
- 「追加の写真撮影は行っておりません」

**Why:** cat02_10 で 4o-Mini が虚偽の eBay ポリシーを根拠にしていた。

### ルール4：FACT TONE-INVARIANT — トーン間で事実情報は不変

POLITE / FRIENDLY / APOLOGY の差は **形式・親密度・語彙のみ**。事実情報（答え・caveat・確認事項）は完全に同一。

- 同じ事実
- 同じ答え
- 同じ caveat
- 違うのは：greeting・縮約形・感嘆符・カジュアル語彙・close phrase

**FRIENDLY は短くしていい、ではない**：「カジュアル ≠ 短い」。POLITEと同量の情報を「より親しみやすい声」で伝える。情報省略は失礼に該当する。

**Why:** cat02 レビューで POLITE/FRIENDLY の差が close phrase ("Best regards" vs "Thanks!") だけになっており、社長から「何がフレンドリーで何が丁寧か気になる」と指摘。

## トーン設計の具体動作（比喩でなく具体ルール）

社長の比喩「良いカフェ店員（ホテルマンほど堅くない・態度悪い店員でもない）」はあくまで **人間→Claude** への伝達手段。**LLM（特に軽量モデル）には抽象比喩は弱く、USE/AVOID 具体語彙リストが効く**。

プロンプト v2.3_baseline_c2 で実装した形：
- POLITE が **所有する語彙**（"Kindly", "Please be advised", "We appreciate" 等）→ FRIENDLY では使用禁止
- FRIENDLY USE 語彙（"Thanks!", "Sure thing", "Got it", "No worries" 等）
- FRIENDLY AVOID 語彙（dismissive: "Look,", "Listen,", "Anyway,", "FYI,"／curt: "Sure.", "Fine.", "Whatever."／minimizer: "Don't worry about it", "It's no big deal"）
- GREETING MIRRORING（バイヤーが Hello なら Hello、Hi なら Hi。FRIENDLY=Hi 強制ではない）

**Why:** LLM軽量モデル（4.1-Mini / 4o-Mini / 5-Mini）は抽象指示より具体トークンルールに強く反応する。「カフェ店員ね、なるほど」と理解する保証はない。
