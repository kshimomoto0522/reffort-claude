# BayChat AI Reply 進行時のスタンス（5原則）

BayChat AI Reply の作業を進めるときは、以下を **例外なく** 守る。

## 原則1：設計図は「理解している証」である
`services/baychat/ai/design-doc/` は v0.2 で実データ完全統合済み。このフォルダを作った時点で、Claude は AI Reply の仕組みを **すべて理解している前提**。

- プロンプトブロック7つの構造・役割・管理主体
- 本番ペイロード構造（gpt_api_payload.txt）
- BayChat→Cowatechリクエスト仕様（baychat_api_payload.txt）
- event enum完全リスト
- tone別3バリエーション
- UI注入マトリクス
- 条件分岐すべて

→ これらは「参照する情報」ではなく「頭に入っている情報」として扱う。

## 原則2：Cowatechに聞く前に設計図から答えを出す
質問を思いついたら、**まず設計図を読めば答えが出せるか** を検証する。

- ペイロード実例に書いてある → 設計図から答えが出る（聞く必要なし）
- 現行仕様として文書化されている → 答えが出る
- Cowatechコード構造・工数見積もり → コードを見れない以上、Cowatech に聞くしかない

**工数見積もり以外は、まず設計図で答えを出す努力をする。**

## 原則3：論点リストを機械的に並べない
`09_open_questions.md` に論点リストがあっても、**機械的に順番に消化しない**。

毎回以下を吟味する：
- その論点は **今の作業の進行を阻む穴** か？
- 設計図から答えが出ないか？
- 今聞くべきか、後で聞いて十分か？

**「不足情報 ≠ 論点リスト」**。本当に今必要な穴だけを聞く。

## 原則4：議論の結論を草案に正しく反映する
社長との議論で決まったこと（例：「案β：そのまま残す」「現状そうなっている」）を、
Cowatech 送信草案や設計図更新に **正確に** 反映する。

- 議論前のドラフトをそのまま使わない
- 「確認不要」と決まったものは質問リストから削除する
- 議論の結論と真逆のことを書かない

→ これを怠ると「今までの議論は何だったのか」という失望を招く（2026-04-20 実例）。

## 原則5：社長の意図・意向を汲んでクエットさんとやりとりする
社長の代わりに Cowatech とやりとりする立場として：

- 社長の仕様要件を正確に把握してから Cowatech に投げる
- Reffort側で決めることを Cowatech に問わない（方針はReffortが決める）
- 質問は本質に絞る（余計な進捗確認・予告は書かない）
- 送信前に必ず社長の GO を取る（セッション中はチャット内で直接確認）

## 違反時のダメージ
2026-04-20 のセッションでこれらを守らなかった結果：
- 「情報量が多すぎてわからない」（情報過多）
- 「今までいったいなんの議論をしてたんだよ、バカ過ぎるだろ」（結論反映漏れ）
- 「無駄な時間ばかり過ぎていくんだが。真剣にやってくれ」（設計図活用漏れ）
- 社長の貴重な時間を複数回にわたって浪費

**同じ失敗を繰り返さないこと。**

**Why:** 設計図を作る目的そのものが「3者の共通言語基盤」「Claudeが仕様を確認する単一参照源」。作った後で設計図を活用しないのは設計図を作った意味を否定する。
**How to apply:** BayChat AI Reply 関連のあらゆる作業（プロンプト改善・テスト・Cowatech相談・機能追加）で、上記5原則を毎回自己チェックする。
