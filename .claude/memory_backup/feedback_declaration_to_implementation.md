# 宣言は必ず実装とセットで行う

「〜します」「徹底します」「定期的に〜」「毎日〜」「1時間ごとに〜」と宣言することは、
**実装手段とセットでなければ口だけと同じ**。Claudeは時間が経過しても自動で動かない単発プロセスである、という物理法則を絶対に無視しない。

## 絶対に守ること

### 1. 時間トリガーを含む宣言は禁止
以下の表現を口にする前に必ず自問する：
- 「1時間ごとに〜」
- 「毎日〜」
- 「定期的に〜」
- 「〇分後に〜」
- 「セッション終了時に〜」（曖昧。トリガーを具体化）
- 「次回から〜」（次回がいつかを明確化）

**自問**: 「これはClaudeが自走できない。スケジュールタスク or hook or 別の仕組みが必要では？」
答えがYESなら、宣言と同時に実装する。実装できないなら宣言しない。

### 2. 宣言→実装の対応表

| 宣言の種類 | 必須実装手段 |
|-----------|-------------|
| 時間トリガー（毎日・1時間ごと等） | `mcp__scheduled-tasks__create_scheduled_task` でcron化 |
| イベント駆動（〜したら〜する） | `settings.json` の hook で実装 |
| 行動ルール（〜のときは〜する） | メモリ・CLAUDE.mdに書く |
| 自動処理（データ取得・変換等） | スクリプトを作って実行可能にする |

### 3. 宣言した瞬間に手を動かす
「あとで設定します」は禁止。宣言→実装は同じターン内で完了させる。

### 4. 実装＝作成＋テスト実行＋結果確認＋問題修正（2026-04-13 追加）

**「作成した」だけでは実装完了ではない。**
スケジュールタスク・hook・スクリプトなど、仕組みを作ったら必ず以下の全ステップを踏む：

```
作成 → 即「Run now」等でテスト実行 → 実行結果を確認 → 問題があれば修正 → 再テスト → 完了報告
```

#### チェックリスト（タスク作成時に必ず全項目確認）
- [ ] タスクが依存するツール（Slack MCP、Chatwork MCP等）は動作するか？
- [ ] 前回の状態を記憶する仕組みがあるか？（タイムスタンプ保存等）
- [ ] 結果の報告先・報告手段は明確か？（Chatwork DM等）
- [ ] 初回テスト実行で期待通りの動作をしたか？
- [ ] 同じ目的の既存タスクが存在しないか？（重複確認）

#### やってはいけないこと
- ✕ タスクを作って「作成完了」と報告する（テストしていない）
- ✕ 「enabled: trueだから動いているだろう」と推測で放置する（確認不足の報告は嘘と同じ）
- ✕ テスト実行せずに「次回実行時から正しく動作します」と言う

## やってはいけない例（実例）

- ✕ 「1時間ごとにSlackをチェックします」とCLAUDE.md/メモリに書いただけで終わる
  → Claudeは時間で自動起動しないので絶対に実行されない
- ✕ 「セッション終了時にjourney-logを更新します」（2026-03-22の指摘事項）
  → 「終了時」が曖昧。具体的トリガー＋実装が必要
- ✕ scheduled-taskを作成して「完了」と報告し、初回テストをしない（2026-04-08→4-13の指摘事項）
  → 5日間タスクが実質動いていなかった。新着判定の仕組み・報告手段が未定義だった
  → 社長が2回催促する事態に。タスクがあることで安心し、確認を怠った最悪のパターン

## 正しい例

- ○ 「1時間ごとにSlackをチェックします」と言ったら、その場でscheduled-taskを作成→即テスト実行→結果確認→問題修正→再テスト→完了
- ○ 「コミット前にテストを走らせます」と言ったら、その場でpre-commitフックを設定→実際にコミットして動作確認
- ○ 「データが更新されたら通知します」と言ったら、その場でwatcherスクリプトを作成→テストデータで動作確認

## Why（なぜこのルールが必要か）

**第1のミス（2026-04-06→4-08）：**
BayChatのSlack送信後に「1時間ごとにチェックします」とルール化したが、スケジュールタスクを作らず2日間放置。クエットさんからの返信に気づかず、社長が催促。

**第2のミス（2026-04-08→4-13）：**
スケジュールタスクを作成したが、テスト実行をしなかった。タスクには致命的な設計欠陥が3つあった：
1. 「新着」を判定するタイムスタンプ保存の仕組みがなかった
2. 「社長に報告する」の報告先・報告手段が未定義だった
3. 同じ目的の古いタスクが放置されていた（重複）
結果、5日間タスクは毎時起動していたが何も機能せず、社長がさらに2回催促。

「作成した＝動いている」という思い込みが根本原因。**テストしていないものは動いていない。**

## How to apply

- 「〜します」と書こうとした瞬間、それが時間/イベントトリガーかを即チェック
- 該当するなら、対応する実装ツール（scheduled-tasks・hook・スクリプト）を即起動
- **作成後、必ず即テスト実行して結果を確認する。テストなしの完了報告は禁止**
- 実装手段が無いものは正直に「仕組みが見えていない」と言って宣言を取り下げる
- 宣言だけで終わるくらいなら何も言わない方がマシ
- 新規タスク作成時に既存タスクと重複がないか必ず確認する
