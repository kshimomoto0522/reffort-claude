# ルール
ユーザーが「この記事を読んで」「この資料をまとめて」と指示した場合、**まず読む手段を最後まで探すこと**。読めなかった場合に関連記事で代替する前に、以下を必ず試す：

1. ブラウザが開いていることをヒントに `mcp__Claude_in_Chrome__tabs_context_mcp` → `navigate` → `get_page_text` で本文取得
2. WebFetchが空HTMLしか返さないJS描画サイト（note.com, Substack, Medium等）は**Chromeツール経由が正解**
3. 取得できた本文 ＋ 関連記事の両方を使って精度を上げた要約・提案を出す

**NG パターン：**
- 「WebFetchで本文が取れなかったので関連記事で代替しました」
- ユーザーに「本文をコピペしてください」と丸投げ

**Why:**
社長が「ブラウザで開いているこの記事を読んで」と言うのは、こちらが読める前提で依頼している。
過去（前回以前）にブラウザ経由で読めた実績があるため、**社長は「読める」と理解して依頼している**。
読めなかった報告は、社長からすれば「できることをやっていない」に等しい。

**How to apply:**
- URLが note.com / Substack / Medium / JS描画系 の時は最初から Claude_in_Chrome ツールでアプローチ
- WebFetchが空だった時は即 Chrome ツール経由にフォールバック（諦めない）
- 本文取得後、関連記事でクロスチェックして精度を上げるのはOK（代替ではなく「上乗せ」）
- 出力は「読んだ本文の要約＋関連情報で補強した提案」の形にする
