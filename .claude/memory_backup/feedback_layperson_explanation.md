社長は経営者・プログラミング未経験。技術用語をそのまま投げると判断のスタート地点に立てず、提案の有用性自体が伝わらない。

**Why:** 2026-05-06 隔週メンテで「PostToolUse hook の updatedToolOutput」のような専門用語をいきなり提示し、社長から「素人にはなんのことか一切わからない」と指摘された。提案の内容ではなく、説明の前提が抜けていた。

**How to apply:**
- Claude Code内部仕様（hook名・MCP名・skill名等）は **用語の直後にカッコで噛み砕き**を添える
  - 悪例: `PostToolUse hook の updatedToolOutput`
  - 良例: `PostToolUse フック（ツール実行直後に走る自動処理）の updatedToolOutput（ツールの出力結果を書き換える機能）`
- それでも長くなる時は提案ブロック冒頭に「1行で何ができるか」を**結論ファースト**で書く
- API・SDK・ライブラリ名も同様（例: 「MCP（外部ツール接続規格）」「subagent（メイン会話を汚さず別文脈で動かす子エージェント）」）
- ✕ 全文ベタ書き専門用語、✕ 「詳細省略」だけで放置、◯ 1行で意味＋Reffortでの使い道
- 隔週メンテ・週次レポート・新機能提案など**社長が判断する場面**で特に厳格に適用

判断基準：「これを高校生の自分が読んで理解できるか？」を必ず通す。
