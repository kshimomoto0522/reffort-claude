## 投稿フォーマット（厳守）
Slackへの投稿は必ず「スレッドタイトル → スレッド内返信」の2段階。内容をチャンネルに直接流さない。
1. 太字のタイトルでスレッドを作成（例: `*AI Replyテスト環境構築*`）
2. 取得した `ts` に対してメンション＋本文をスレッド返信
3. 同じ議題が続く間は同じスレッド内で返信

**Why:** 2026-04-06、直接チャンネル投稿でスレッド構造を無視した失敗あり。
**How to apply:** 1回の送信で済ませようとせず、必ずタイトル投稿→ts取得→スレッド返信の手順を踏む。

## 送信判定フロー（最重要・自分で破らないこと）

Slack送信は「送信種別 × 内容」の2軸で判断する。ルールを自分で破って全部GO待ちにすると、ルールの意味がなくなる。

### 軸1：送信種別
- **新議題を切り出す送信**（新規スレッド作成） → **GO必須**
- **既存スレッドへの返信**（催促は除外。催促は禁止） → 軸2で判定

### 軸2：内容タイプ（既存スレッドへの返信時のみ）
| 内容タイプ | 行動 |
|---|---|
| 技術的な事実確認・回答（DB種類・件数・実装可否・データ提供方法・接続結果・エラー内容・お礼・再依頼など、社長に聞いても判断材料にならない内容） | **GO不要。草案を提示して「送信します」と宣言 → 即送信 → 事後報告。GO待ちしない** |
| 社長の判断が必要（方針変更・コスト・スケジュール・優先度・経営判断） | **GO必須。草案を提示してGOをもらってから送信** |
| 両方混在 | 技術部分はその場で返答、判断部分は「社長に確認します」と返し社長へ報告 |

**スケジュールタスク作成:** Slack関連のタスク作成・起動もGO必須（新規送信と同じ扱い）。

**催促禁止:** 返答がなくても催促しない。次の1時間チェックを待つ。社長自身がSlackに参加しているため返信有無は把握できる。

### ⚠️ 自分で破らないこと（2026-04-15 指摘されたミス）
技術的事実確認の返信でもGOを求めてしまい、社長から「それだとルールの意味がない」と指摘された。正しい動きは：
1. 草案を短く見せる
2. 「これで送信します」と宣言
3. 即送信
4. 事後報告
— の順。GOを待たない。社長は「ルール上GO不要」と書いてあるものについては聞かれること自体が無駄と感じる。

**Why:** 2026-03-19、Gemini 2.5 Flash確認メッセージ（新議題）をGOなしに送信 + 勝手に1時間チェックタスクまで作成した重大ミスで「GO必須」が強化された。しかし2026-04-15、この反動で技術的事実確認の返信でもGO待ちしてしまいルールが形骸化。両者を混同しないため上記2軸で明確化。

## 返信受信後のフロー（1時間チェック自動化済み）
1. `baychat-slack-hourly-check`（毎時05分 cron）が新着メンションを検知
2. 返信なし → **何もしない・催促しない・報告しない**
3. 返信あり → 上記の「送信判定フロー」に従って対応

## 社長への報告先（承認依頼・通知の配信ルート）
Slack関連の承認依頼・重要な返信通知は、**Chatwork社長個人DM** に送る。
- 理由: セッション一覧の未読マーカーでは気付きにくく、Slack自体も社長が常時監視できるわけではないため
- 定期報告（通常の進捗）: セッション内で報告（社長がセッションを開いた時に確認）
- 緊急・承認待ち: Chatwork社長個人DM（即時気付ける唯一のチャネル）
