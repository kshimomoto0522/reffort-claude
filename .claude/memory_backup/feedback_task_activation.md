## ルール1: 新規タスクはアクティベートテスト必須
スケジュールタスクを新規作成しただけでは、ツール権限（Chatwork API・Chrome操作等）が未承認。無人実行時に承認ダイアログで停止し何も実行されない。作成後すぐにテスト実行して承認を通す。

## ルール2: 繰り返し処理はキュー方式で1タスクを使い回す
同じ種類の処理（例：メンバー削除）を複数人に対して行う場合、人ごとにタスクを作るのではなく、キューファイル＋cronタスク1つで運用する。承認は1回で済み、社長はキューに追加するだけ。

## ルール3: 社長の意図を汲んで自動化を提案する
社長が「自動でやってほしい」と言ったら、その先の運用（次の人の処理、キューの管理、エラー時の対応）まで含めて自動化を設計する。「1回だけ動く仕組み」ではなく「繰り返し使える仕組み」を最初から提案する。これは社長に指摘される前にやるべきこと。

**Why:**
2026-04-13にCampersメンバー削除タスクで複数の失敗。承認未取得による不発、1人ずつタスクを手動作成する非効率な設計、社長の「伝えたら自動」という意図を汲めなかった。

**How to apply:**
- 新規タスク → アクティベートテスト（テスト用プロンプトで即実行→承認→本番プロンプトに更新）
- 繰り返し処理 → キューファイル + cron方式を最初から提案
- ディスパッチャー停止時 → Claude Code再起動で復旧
- Campersメンバー削除 → `education/campers/removal-queue.json` + 毎朝5:00 cronタスク `campers-member-removal-goto`
