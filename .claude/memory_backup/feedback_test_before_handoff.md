## ルール

ツール・スクリプト・コード・デプロイ・設定変更など、**何かしら実装・修正したものを社長に渡す時は、こちらで「動作テスト」まで完了させてから依頼する**。

「できました」「修正版を配置しました」「実行してください」と報告する前に、**最低限の起動テスト＝処理は走らせなくても、エラーで即死しないか確認**するのが必須。

## なぜ

社長の時間を奪う。手元で起動 → 一瞬で消える → 「動かない」報告 → Claudeが調査 → 再ビルド → 再依頼、というサイクルは時間も信頼も損なう。**Claudeが先回りで5分使えば防げる**こと。

## 違反履歴

- 2026-04-30 ASICSツール（旧exe）修正・再ビルド・本番デプロイ後「実行してください」と社長に依頼
  - → 社長がダブルクリック → 黒画面1秒で消える（`charset_normalizer` 等の隠れ依存をPyInstallerが見落としていた）
  - → 起動テストすらしていなかった
  - → ロールバック・再ビルド・再デプロイで時間ロス

## How to apply

### 必須テスト（最低ライン）

| 種類 | テスト内容 |
|---|---|
| **コンパイル系（exe・dll等）** | 必ずビルド後に**起動テスト**。クリーン環境で起動し、importフェーズが通過するか確認（クラッシュしない・config不足エラーまで到達するなら起動はOK） |
| **Pythonスクリプト** | `python script.py --help` 等で起動可能か、または短時間実行でimportとmain()到達まで確認 |
| **GAS（Apps Script）** | clasp push後に `clasp run` または手動でtrigger設定確認・初回1回はテスト実行 |
| **設定ファイル変更（settings.json等）** | 設定が正しく読み込まれるか、Claude/CLI側で再起動して反映確認 |
| **スプレッドシート連携（gspread・Sheets API）** | 認証通過 + 1セル読み込み or 1セル書き込みでAPI接続確認 |
| **タスクスケジューラ・cron** | 登録後に「Run now」で1回手動実行・成功確認 |
| **本番デプロイ前のロールバック手順** | 失敗時に1コマンドで戻せる状態にしてから配置 |

### NGパターン

- ビルド成功＝OKと判断して即依頼
- 構文チェック（py_compile / lint）だけで「動作確認しました」と言う
- AST解析で関数定義の存在だけ確認して動作確認したことにする
- 自分で起動テストせず社長に「実行してください」と投げる

### OKパターン

- ビルド後にクリーンディレクトリで起動 → 「configが見つからない」エラーまで到達 → import OK確認 → 配置 → 依頼
- 「テスト時のログを社長に提示してから」依頼
- 失敗ケースを潰してから「動作確認できる状態にしました、実行お願いします」

### 例外（テストできない場合）

物理アクセス・社長のローカル環境固有・eBay/Slack等の本番副作用が避けられない、等で**Claudeでテスト不可能**な場合は：

1. 「Claude側でこれ以上テストできない理由」を**先に**伝える
2. 「失敗時の即時ロールバック手順」を**事前に**示す
3. その上で社長にテスト実行を依頼

無言で投げて出鼻くじかれるのが**最悪パターン**。理由を添えて依頼するのは正常運用。
