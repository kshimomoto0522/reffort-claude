## 0. 今セッションのサマリ

### v9 並列化は中止
- DECODO Pay As You Go コスト試算: 約 19,000円/月（ASICS のみ・他サイト追加で倍々）
- AdsPower(Onitsuka) はスタッフ仕入れ業務と共有 → 16h/日連続稼働でロック競合
- → コスト過大＋運用支障で **ローカル単独運用維持**を採用
- v9 関連コード（`worker_split.py` `adspower_driver.py` `run_parallel.bat`）は将来再利用可能性ありで残置

### v8 → v8+補正版 への改修と本番投入
v9 開発中に発見した構造的バグ + 運用中に判明した削除エラー扱いを v8 ベースに反映：

| 修正 | 内容 |
|---|---|
| **論点①** | `identify_retry_targets` で URL空欄+ItemIDあり+未処理の行を Mod E 対象として拾うよう改修 → 過去 exe で素通り→永遠に未処理放置の構造的問題を解消 |
| **論点②** | Policy違反表示を在庫状況とエラー情報で分離（旧:在庫状況=`Policy違反 (X/在Y)` → 新:在庫状況=`X/在Y`, エラー情報=`Policy違反`）。`compute_inventory_status` 戻り値を 3要素タプルに拡張 |
| **論点③** | EndItem ErrorCode `1047` (auction has been closed) を **成功扱い** に分岐追加 → eBay 側で既に終了済の場合に Q列・行A〜S列を自動クリアするよう改善（社長指摘 S02754 のケース対応） |

### 待機時間 240→210秒に短縮（試行）
社長判断で 240秒 → **210秒**に短縮。1周40h → 35h（12.5%短縮見込み）。
**戻し条件**：1周のうち 5件連続Bot検出（30分休憩）が **2回以上**発生したら 240秒に戻す。

## 2. コード変更箇所（`asics_master_work/scrape_data.py`）

### 2.1 compute_inventory_status (line 350〜)
- 戻り値: 2要素 → **3要素タプル `(status_str, should_process, error_info)`**
- Policy違反時: `("X/在Y", True, "Policy違反")` を返す（旧: `("Policy違反 (X/在Y)", True)`）

### 2.2 process_one_item (line 1266付近)
- `compute_inventory_status` の呼び出しを 3要素受け取りに変更
- `write_one_item` の error 引数に `error_info` を渡す

### 2.3 Mod E（URL空欄処理・line 1104付近）
- Policy違反分岐: `status_e=f"0/在{z_count_e}"`、`error_info_e='Policy違反'` に分離
- write_one_item に `error_info_e` を渡す

### 2.4 identify_retry_targets (line 1037〜)
- `'ebay出品\nItemID'` と `'更新日時'` を追加取得
- 永続エラー判定に `if err == 'Policy違反': continue` を追加（新形式対応）
- **新規**: `(not url) and item_id and (not update_dt)` → URL空欄+ItemIDあり+未処理 を Mod E 対象として拾う

### 2.5 end_item_via_api (line 778〜)
- **新規**: `if error_code == '1047' or 'auction has been closed' in short_msg.lower(): return {'status': 'ended', ...}`
- eBay 側で既に終了済の出品も成功扱いで自動クリーンアップ

## 6. 運用ルール追加（今セッション気づき）

### 本番 exe を一時停止する作業を行うとき
社長が出品削除予約（Q列チェック）を入れたタイミングで Claude が exe を停止していると、削除キューの検知が止まる。今後 Claude が本番 exe を停止する作業をする時は：

1. **作業開始前**：社長に「これから本番 exe を停止します。停止中の Q列チェックは検知されません」と一言宣言
2. **作業完了後**：起動直後の削除キュー処理が完了するまで観察し、結果を社長に報告
3. **長時間停止になる場合**：その旨を Chatwork 個人 DM に通知（必要なら）
