## 第3章: 真因特定（肥大化・hook・ノイズのファクト）

**数値で語る根本原因**。主観を排除し、measurable な事実で立証。

### 真因3本柱

#### 柱1: カスケード・コンテキスト爆発
- 作業フォルダで親4層のCLAUDE.mdが自動ロード
- `services/baychat/ai/` で作業時に 572行+rules150行+MEMORY.md61行 = 初期段階で **5,000〜8,000トークン消費**

#### 柱2: effort_booster過アグレッシブ
- `COMPLEX_KEYWORDS` に日常語（分析・検証・判断・バグ・エラー・方針・比較・改善・なぜ・原因）が含まれていた
- 社長の普通の発話で ultrathink 自動注入 → **トークン倍増・レイテンシ増**

#### 柱3: ノイズファイル氾濫
- `commerce/ebay/tools/` 配下 77個の中間生成物
- `scrape_data.exe_extracted/` **190MB** がgit追跡下
- Claudeのファイル探索で誤ヒット・コスト増

## 第6章: 永続仕組み（持続可能性4点セット）

**肥大化の再発を物理的に防ぐ仕組み**。

### 4点セット

#### ① archive/ フォルダで完了情報を退避
- 全部門に `archive/` 配下を新設
- Claudeは**原則読まない**（明示指示時のみ）
- 情報資産保全とコンテキスト軽量化を両立

#### ② biweekly-claude-maintenance 隔週タスク
- **第1・第3月曜 10:00** 自動実行
- スクリプト: `management/md-audit/biweekly_maintenance.py`
- 機能: 定量計測（CLAUDE.md行数・memory・settings・effort_booster発火）＋Chatwork個人DM通知

#### ③ 各部門 index.md で退避ファイル一覧を管理
- 「常時ロード / 必要時ロード / archive/参照なし」の3層で可視化
- 新規退避時、更新すべきは index.md 1ファイルだけ → 探索コスト常に一定

#### ④ `/隔週メンテナンス` スラッシュコマンド
- Chatwork DM読み込み → 5層調査 → 松竹梅提案 → 社長判断 → 実行 → 結果再送信
- Claude Code 1コマンドで改善サイクルが回る半自動フロー
- 全過程を `session-logs/YYYY-MM-DD.md` に蓄積（コンテンツ素材化）

## 第7章: 受講生への提言

**Campers視点のまとめ**。

### AI運営は「1回セットアップして終わり」ではない
- 2週間ごとに点検する習慣を持つ
- 肥大化の兆候（レスポンス遅延・嘘・バカな回答）は早期警告

### 失敗を隠さず記録する → 将来のネタになる
- `session-logs/` に生のまま蓄積
- 成功事例だけでなく、軌道修正・逆提案・失敗も書く

### 「AI × 事業」は運用こそが本番
- 導入: 誰でもできる
- 運用: 肥大化との戦い・進化への追随
- 差別化: メンテナンスの質が長期差になる

### 読者への持ち帰り課題
- 自分のCLAUDE.mdを今すぐ `wc -l` してみよう
- 200行超えていたら、Progressive Disclosure を試してみよう
- 隔週の点検タスクを作ってみよう
