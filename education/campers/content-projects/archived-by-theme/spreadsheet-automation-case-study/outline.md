# コンテンツ化骨子：スプレッドシート自動化（Campers / コンサル教材想定）

> 受講生は「プログラミング未経験〜半経験」の中小事業者。「AIにスプレッドシート任せたい」がゴール。
> 本骨子は記事・動画・スクール教材化する際の **章構成テンプレート**。session-logs/ に溜まる生記録から該当章へ素材を切り出して使う。

---

## 第1章：なぜAIにスプレッドシートを任せるのか

- 中小事業者の業務はほぼスプレッドシートに集約されている
- 手作業の限界：転記ミス・更新忘れ・属人化
- 「AIにやらせる」と言っても、やり方を間違えるとすぐ詰む
- → 本講座では「3つの型」を使い分けるロードマップを提示

## 第2章：3つの型を見極める（最重要）

### 型A：シート上で動くロジック（Apps Script）
- 例：シート開いた瞬間メニュー表示／行追加トリガー／時間トリガーで API叩いて自動入力
- 適例：仕入管理表（毎朝 eBay APIから新規オーダーを取得→反映）

### 型B：外部からシートを読み書き
- 例：Pythonの朝バッチで集計→シートに書込／在庫管理ツールがシート更新
- 適例：週次レポート（Pythonで Excel生成＋Sheets書込→ Chatworkに配信）

### 型C：シートのUI操作（最終手段）
- 例：管理画面でしか操作不可のUI設定／GASエディタ自体を触る
- 適例：Campersグループの権限OFF/ON操作

## 第3章：型Aの実装（Apps Script + clasp）

- なぜ「Monaco editorに手動でCtrl+V」運用は破綻するのか
- clasp（Google公式CLI）を使うと何が変わるか
- セットアップ手順（`clasp login` → `clasp clone` → `clasp push`）
- 1つのGoogleアカウントで全Apps Scriptを管理する仕組み
- 実例：仕入管理表（76KB・1780行）の移行ステップ

## 第4章：型Bの実装（Google Sheets API + Service Account）

- Service Accountとは（人間アカウントとの違い）
- なぜサービスアカウントが運用上強いか（ログイン切れがない・トークン管理不要）
- セットアップ手順（GCPでサービスアカウント作成 → JSONキー → スプレッドシート共有）
- Pythonでの読み書き（gspread / google-api-python-client）
- 実例：eBay週次レポートの GSheets書込

## 第5章：型Cの実装（Playwright + cookieセッション）

- なぜ Claude in Chrome MCP が無人実行で詰まるか（Anthropic既知バグ #30356/#47180）
- Playwright + storage_state で「初回手動ログイン → 以降cookieで自動」の仕組み
- 起動経路：Claude scheduled-task ではなく **Windowsタスクスケジューラ直接** が確実
- 失敗時の安全装置（権限変更を必ず元に戻す・対象以外を絶対消さない）
- 実例：Campersメンバー削除（権限OFF→API削除→ON×2部屋＋全体連絡API削除）

## 第6章：間違えた選択肢の具体例（失敗ケース）

- **NG例1**: 型Aで済むのに Playwright で Monaco editor を操作しようとする
  - bot検知・Monacoの不安定さで遠回り
- **NG例2**: 型Bで済むのに GAS を新規作成
  - GASは制約が多い（実行時間6分・同時実行制限・トリガー回数制限）
- **NG例3**: 型Cが必要なのに API で無理やり片付けようとする
  - Chatwork公式APIで「メッセージ送信OFF」できないのに「APIでスキップ」してしまい本番事故

## 第8章：運用継続のチェックポイント

- 認証情報の管理（.env / Service Account JSON / clasp認証）
- セッション切れ・トークン失効への対処
- 失敗時に必ず DM/通知で気付ける構造を仕込む
- ログをどこに残すか・どう見るか
