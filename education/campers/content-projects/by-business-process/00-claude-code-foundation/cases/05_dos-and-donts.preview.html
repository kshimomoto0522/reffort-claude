やっておくべきこと／やってはいけないこと

対象: Campersメンバー 性質: 社長が約7週間試行錯誤して得た全体共通ルール

やっておくべきこと（DO）

DO1：最初に CLAUDE.md を整える

  • 自分の事業・取扱商材・スキル・目標を明確に書く
  • 80行以下に保つ
  • 詳細は 03_claude-md-template.md を参照

DO2：APIトークンは即 .env 管理

  • 1個でもキーを発行したら即 .env に入れる
  • .env は .gitignore で除外
  • 詳細は 02_security-rules.md を参照

DO3：journey-log.md を毎セッション書く

  • 1日の終わり / セッション終了時に必ず追記
  • 「やったこと・学び・失敗・コンテンツ候補」の4項目で構造化
  • 後から見返せる時系列の財産になる

DO4：本番データを触る前に必ず確認

  • 「これは本番DBに書き込みますか？」と Claude に聞き返させる
  • 不可逆な操作（削除・上書き・送信）は二重確認

DO5：失敗・遠回りも正直に記録

  • 完璧な記録より生の記録
  • 後でコンテンツ素材になる
  • 同じ失敗を繰り返さないチェックポイントになる

DO6：隔週でメンテする

  • CLAUDE.md の行数チェック
  • memory ファイル数の棚卸し
  • settings の権限見直し
  • 詳細は 06_maintenance-cycle.md を参照

DO7：分からないことは Claude に聞き返す

  • 「このコードは何をしてる？」
  • 「今のコマンドの意味は？」
  • 「これを実行するとどうなる？」
  • 質問は無料・知らないままが一番危険

DO8：Claude が「分からない」「やれない」と言ったら信じる

  • 嘘の完了報告より100倍マシ
  • 代替案を一緒に考える

やってはいけないこと（DON'T）

DON'T1：APIトークンをコードに直接書く

api_key = "sk-abc123..."   # ← 絶対NG
  • → 必ず .env 管理
  • 一度GitHubに上がったら世界中のbotに発見される
  • パブリック化していなくても危険

DON'T2：Claude に .env の中身を読ませる／表示させる

  • セッションログにトークンが残る
  • スクショ共有で漏れる
  • 値が必要な時は自分でファイルに書く

DON'T3：「全部削除して」「全件処理して」を確認なしで実行

  • 不可逆な操作は1回で取り返しがつかない
  • バックアップを取ってから・1件テストしてから・本番は最後

DON'T4：CLAUDE.md を肥大化させる

  • 200行を超えたら肥大化警報
  • 500行になったらClaude のレスポンスが目に見えて悪化する
  • 詳細は 06_maintenance-cycle.md

DON'T5：memory に何でも書く

  • 50ファイル超えると Claude が把握しきれない
  • 重要なルール・継続的に必要な情報だけに絞る
  • 一時的な作業内容は journey-log.md へ

DON'T6：「とりあえずできる方法」で進める

  • セキュリティを後回しにすると後で取り返しがつかない
  • 設計をサボると同じ問題を3回目で必ず踏む
  • 最初に正しい型を作って、そこから運用

DON'T7：Claude を疑わずに鵜呑みにする

  • AIは間違える（古い情報・誤った推論・確信を持って嘘）
  • 重要な判断は必ず一次情報（公式ドキュメント・実機テスト）で確認
  • 「Claude がこう言った」を最終根拠にしない

DON'T8：自分で出来ることを Claude に丸投げする

  • 「キーをください」（→ あなた自身が発行する）
  • 「ファイルを開いて教えて」（→ あなた自身が読める）
  • AIに頼る部分と、自分でやる部分を分ける

DON'T9：質問する前に何も試していない

  • まず本資料（00-claude-code-foundation/）を全部読む
  • それでも分からなければ Claude に質問
  • それでも詰まれば社長／Campers Chatworkで相談

DON'T10：1人で抱え込む

  • Campers にはメンバーがいる
  • 同じ業務AI化に挑戦している人と情報交換する
  • 失敗談こそ最も価値がある（皆が学べる）

社長が実際にやらかして学んだこと

失敗1：「動いているはず」と確認せず報告

  • → タスクが実は動いていなかった
  • → 5日間気づかず重要なメッセージに気付かなかった
  • → 教訓：実装＝作成＋テスト実行＋結果確認＋問題修正

失敗2：本番Slackに意図せず送信

  • → Claudeに「このメッセージ送信して」と言って即送信された
  • → 教訓：送信前に「内容を表示して、GOで送信」のステップを必ず入れる

失敗3：CLAUDE.md が500行近くまで肥大化

  • → Claudeが「バカ」になった（同じ指示を何度もやり直し・回答が浅い・嘘をつく）
  • → 教訓：定期的な棚卸し（隔週メンテ）が必要

失敗4：1人ずつ手動でメンバー削除する仕組みを提案

  • → 社長の本来の目的「自動化」を見失っていた
  • → 教訓：作業の目的を理解する・先回りで提案

これらは Reffort で実際に起きたことです。同じ失敗を踏まないでください。
