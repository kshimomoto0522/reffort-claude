# 03. 無在庫管理ツール — スクレイピング型在庫追跡のAI化

> このフォルダは「無在庫出品の在庫・価格をリアルタイム追跡」する業務のAI化事例を集める棚です。

---

## この業務段階で何ができるか（メンバー向け）

### 課題（メンバーが抱えがちな悩み）
- 無在庫出品で「仕入元の在庫切れ」を見逃して欠品クレーム
- 仕入元の価格変動に追従できず利益消失 or 赤字
- メルカリ等のフリマで「売り切れ」を即座に検知できない
- 数百〜数千SKUを手動チェックは現実的に不可能

### Claude Code でできること
- **仕入元サイト（メルカリ・公式EC・並行輸入サイト等）を24時間自動巡回**
- **在庫切れ／価格変動を即座に検知 → eBay出品をOFF or 価格更新**
- **Bot検出への対策**（stealth設定・Firefox session・ヘッドレス回避）
- **失敗時の安全装置**（誤って大量取り下げしないチェックポイント）

## つまづきポイント（社長の体験：ASICS無在庫管理）

### 失敗1：Bot検出がスクレイピングを止める
- ASICS の Akamai Bot 検出により完全ブロック
- 通常の Selenium / Puppeteer では即検出される
- 対策：stealth設定強化・Firefox session継続・ヘッドレス回避

### 失敗2：「監視すれば監視するほどブロックされる」パラドックス
- 自動監視の頻度を上げる → Bot検出が加速
- 対策：適切な間隔（1時間以上）・User-Agent ランダム化・並列化を控える

### 失敗3：旧ツールの環境破壊
- 新ツールに切り替え時に旧ツールを止めず両方稼働 → 競合
- シート名ハードコード（例：【ASICS】→ eBay在庫調整 への移行漏れ）
- 対策：移行は段階的に・旧ツールは即停止

### 失敗4：「動いているはず」報告
- exe が起動しているのを確認しただけで「稼働中」と報告
- 実は内部でエラーが連発していた
- 対策：実装＝作成＋テスト実行＋結果確認＋問題修正

## メンバーへの含意

- **あなたの仕入元サイトでも同じ手法は使える**（メルカリ・ラクマ・海外公式EC等）
- **Bot検出は宿命**（最初は検出される前提で・stealth設定必須）
- **完璧を目指さない**（80%取れれば十分・100%は逆に検出される）
- **エラーハンドリングは初日に**（失敗を検知して通知する仕組みを最初に）
