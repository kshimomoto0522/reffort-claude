在庫管理ツール（無在庫スクレイピング型）

業務段階: 03-無在庫管理 状況: 社長運用中（ASICS／拡張可能） 元素材: journey-log.md L200-237 / L494-533 / L703-741 / L744-771

何をやったか（目的・背景）

仕入元サイト（ASICS公式EC等）から商品情報・在庫・価格を自動巡回し、自社eBay出品にリアルタイム反映する仕組み。

無在庫出品の最大の弱点「仕入元在庫切れに気付けない → 欠品クレーム」を解消するために構築。

社長は新品スニーカーを扱うため ASICS／adidas／NIKE 等の公式EC・並行輸入サイトを対象としているが、メンバーの取扱商材によって対象サイトを差し替えれば同じ仕組みで運用可能。

取り組みのプロセス（時系列要点）

v1初版（〜2026-03-22）

  • ChromeDriver直接利用
  • Akamai Bot 検出により完全ブロック
  • 安定稼働は数時間で停止

v2.2 安定（2026-03-22）

  • stealth設定強化（playwright-stealth相当）
  • Firefox session継続
  • ヘッドレス回避
  • 240秒/件・成功率約55%

v2.3 並列化試行 → 失敗（2026-03-26）

  • 並列処理で速度を上げようとした
  • Bot検出が逆に加速・失敗率上昇
  • Firefox 再起動を頻繁にかけたことも検出材料に

v2.3 修正版（2026-03-27）

  • Firefox再起動を廃止・累計失敗カウントを撤廃
  • 直列処理を維持
  • 84件/14h・処理量9倍（成功率は45%だが処理量が上がった）

旧ツール復旧（2026-03-30）

  • 並行運用していた旧ツールがシート名ハードコードで動作停止
  • 「【ASICS】」→「eBay在庫調整」へのシート名変更を反映漏れ
  • 修正完了で旧ツールも復旧

つまづいたこと（失敗・遠回り）

失敗1：Bot検出 = IPブロックと誤認

  • IPアドレスが原因と思い込み、IP変更を繰り返す
  • 実態は「フィンガープリント検出」（ブラウザの指紋検出）
  • 教訓：まず検出メカニズムを正しく理解する

失敗2：「動いているはず」報告

  • exe ファイルが起動しているのを確認しただけで「稼働中」と社長に報告
  • 内部では連続エラーが発生していた
  • 教訓：実装＝作成＋テスト実行＋結果確認＋問題修正

失敗3：監視がBot検出を加速させるパラドックス

  • 自動監視の頻度を上げた → Bot検出に引っかかりやすくなる
  • 「監視すればするほど止められる」というジレンマ
  • 教訓：適切な間隔（1時間以上）・User-Agent ランダム化・並列を控える

失敗4：旧ツール環境破壊

  • 新ツールへの移行時、旧ツールを停止せず両方稼働
  • シート名の変更を旧ツール側に反映していない
  • 教訓：移行は段階的に・旧ツールは即停止

学んだこと（教訓）

  • スクレイピングはBot検出との戦い：技術的優位は数ヶ月で崩れる前提で運用設計
  • stealth設定は最初に：後付けは難しい
  • 並列化は最後の手段：直列で安定を確保してから
  • エラーハンドリング・通知が必須：失敗を検知できないと「動いているはず」になる
  • 完璧を目指さない：80%取れれば十分・100%取ろうとして詰まる

メンバーへの含意（応用ポイント） ★

あなたの仕入元でも同じ手法は使える

  • メルカリ・ラクマ・Yahoo!オク
  • 公式EC（NIKE / adidas / Uniqlo / Rakuten 等）
  • 海外サイト（並行輸入）

必要な前提

  • 仕入元サイトのURL構造を理解できる
  • Python or Node.js で Playwright を動かせる
  • .env で認証情報を管理できる

最初の1個を作る手順

  1. 監視したい商品を10個選ぶ（多くしすぎない）
  2. それらのURLをスプレッドシートに記録
  3. Claude に「このURLリストを毎日1回巡回して在庫・価格をシート更新するスクリプト書いて」と依頼
  4. 手動でテスト → エラー対処 → 1日運用 → 1週間運用
  5. 安定したら対象商品を増やす

やってはいけないこと

  • ❌ いきなり1000商品の自動巡回（Bot検出で即停止）
  • ❌ 1分間隔の監視（過剰アクセス）
  • ❌ stealth設定なしのSelenium（即検出）
  • ❌ エラー通知なしの放置運用（壊れていても気付かない）
