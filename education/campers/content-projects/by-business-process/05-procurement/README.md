# 05. 仕入管理 — 仕入・発注業務のAI化

> このフォルダは「仕入先への発注・入荷管理」業務のAI化事例を集める棚です。

---

## この業務段階で何ができるか（メンバー向け）

### 課題（メンバーが抱えがちな悩み）
- 売れた商品の仕入先・発注タイミングを毎回手動確認
- スタッフへの仕入指示が伝わりにくい・ミスが起きる
- 仕入先別の発注履歴・条件管理が煩雑
- eBayの注文情報と仕入管理表の連携が手動で時間取られる
- Ship by Date を計算間違えて発送遅延

### Claude Code でできること
- **eBay受注 → 仕入指示の自動生成**（GAS / Sheets API / Fulfillment API）
- **仕入先別の自動振り分け**（商品ごとに仕入先を判定）
- **Ship by Date の正確計算**（eBay公式の発送期限を API から取得）
- **キャンセル・返金の自動検知**（無駄な仕入を防止）
- **重複発注防止**（既発注を3層チェック）

## 社長の体験：仕入管理表GAS（要点）

### 何をやったか
- スプレッドシート（仕入管理表）に GAS（Google Apps Script）を組み込み
- eBay の Fulfillment API から受注情報を自動取得
- 商品ごとに仕入先を自動判定 → 行追加
- スタッフが見やすい色分け・並び順を自動メンテ

### 取り組みのプロセス
- v1: Trading API（古いSOAP API）で実装 → 制限多く動作不安定
- v2: Fulfillment API（REST API）に移行 → 動作安定・OAuth2導入
- v3: clasp 導入で開発効率向上（ローカルエディタで開発・GAS自動デプロイ）
- スタッフ運用フィードバックで4点修正（多言語サイズ・Width機能・キャンセル検知・Refunded蘇生）

### つまづいたこと
- Z除去（マイナーなSKU処理）の副作用で重複バグ発生
- Monaco editor 手動 Ctrl+V 運用は信頼性低（途中から clasp に切替）
- 複数ファイル構成の管理が複雑

### 学んだこと
- **Fulfillment API のほうが Trading API より安定**（OAuth2は最初の壁・以降は楽）
- **重複チェックは3層必要**（同SKU・同日付・同BuyerID）
- **clasp 必須**（Monaco editor 手動運用は破綻する）
- **スタッフフィードバックが最も価値ある改善源**

### 数値 Before/After
- 手動運用：1注文あたり 2〜3分の手作業
- GAS自動化：1注文あたり 0秒（完全自動）
- スタッフの仕入忘れ：月数件 → ゼロ

## メンバーへの含意

- **仕入管理は AI化の効果が最も実感しやすい業務**（手作業が多く・ルーチンが明確）
- **既存スプレッドシートを活かせる**（GAS なら手動運用と並走可能）
- **OAuth2は最初の壁だが越える価値あり**（一度設定すれば以降は楽）
- **スタッフを巻き込め**（運用フィードバックが改善の宝）
