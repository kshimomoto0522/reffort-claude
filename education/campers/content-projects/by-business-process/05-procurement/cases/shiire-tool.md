# 仕入管理表GAS（Trading→Fulfillment API移行・clasp化）

> 業務段階: 05-仕入管理
> 状況: 社長運用継続中（2026-04-24 Fulfillment API移行・clasp化完了）
> 元素材: journey-log.md L1040-1075 / L1519-1584 / L1965-2044

---

## 何をやったか（目的・背景）

eBay受注情報から自動で仕入指示を生成し、スタッフに「何を・いつまでに・どこから仕入れるか」を伝える Google スプレッドシート + GAS（Google Apps Script）の仕組み。

社長の運用では、外注スタッフ3名（本間・清水・佐々木）が毎日この表を見て仕入を実行する。

---

## 取り組みのプロセス（時系列要点）

### v1 Trading API実装（〜2026-04）
- 古い SOAP API（Trading API）で実装
- 制限が多く動作不安定
- 認証は Auth'n'Auth トークン

### 機能追加3件（2026-04-13）
- 有在庫判定（自社在庫があるか）
- E列結合（複数情報を1セルに統合）
- 赤文字化（要確認案件のハイライト）
- Z除去による重複反映バグ発覚

### スタッフ指摘4点修正（2026-04-21）
- 多言語サイズ表記対応（US/UK/EU）
- Width 機能追加
- キャンセル検知
- Refunded 蘇生（返金された注文を仕入リストから削除）
- 重複3層防御導入（同SKU・同日付・同BuyerID）

### Fulfillment API移行（2026-04-24）
- Trading API → Fulfillment API（REST API）
- OAuth2 認証導入
- Ship by Date を正確化（祝日テーブル不要・APIから取得）
- ミズノ型番10桁化対応
- clasp 導入（ローカルエディタで開発・自動デプロイ）

## つまづいたこと（失敗・遠回り）

### 失敗1：Z除去（マイナーSKU処理）の副作用
- Z付きSKUを除去するロジックを入れたら、本来別商品扱いの「型番A-Z」と「型番A」が同一視された
- 重複反映バグ発生
- 教訓：**正規化は副作用を全パターン考える**

### 失敗2：Monaco editor 手動 Ctrl+V 運用の限界
- Apps Script のエディタ（Monaco）に手動でコードをペーストする運用は崩壊した
- ペースト漏れ・古いコード残存・差分管理不可
- 教訓：**clasp 導入が正解**（Google公式CLI・ローカル開発・自動デプロイ）

### 失敗3：複数ファイル構成の複雑化
- 機能追加のたびにファイル分割
- 5ファイル以上になると依存関係が見えなくなる
- 教訓：**設計を最初にしっかり**・後付けは2ファイルまで

### 失敗4：スタッフ運用フィードバックを後から拾う
- 機能追加時にスタッフに事前確認しなかった
- 結果4点もの修正リクエストが運用後に来た
- 教訓：**スタッフ運用フィードバックは仕様策定の初日に取る**

---

## 学んだこと（教訓）

- **Fulfillment API のほうが Trading API より安定**：OAuth2 は最初の壁だが越える価値あり
- **重複チェックは3層必要**：単一キーでは不十分
- **clasp 必須**：Monaco editor 手動運用は破綻する
- **スタッフフィードバックが最も価値ある改善源**：運用してみて初めて分かる
- **Ship by Date は eBay公式 API から取得**：祝日テーブル等の自前実装は不要

## メンバーへの含意（応用ポイント） ★

### あなたの仕入フローでも同じ仕組みは作れる
- 商材問わず「受注 → 仕入指示 → 仕入完了 → 発送」のフローは共通
- スタッフがいる人ほど効果大（指示伝達のミス・遅延が消える）

### 必要な前提
- Google アカウント（スプレッドシート）
- Apps Script の基礎知識（Claude Code が書いてくれるが、概念は理解必須）
- eBay Developer プログラム登録（OAuth2 用）
- スタッフがスプレッドシートを毎日見る運用

### 最初の1個を作る手順
1. 既存の仕入管理表（または新規作成）にどんな項目が必要か洗い出し
2. eBay受注情報のうち何を反映するか決める（最低限：注文日・SKU・サイズ・送料・Ship by Date）
3. Claude Code に「eBay Fulfillment API から受注を取得して、このシート構造で行追加するGAS書いて」と依頼
4. テスト用シートで動作確認
5. スタッフに使ってもらってフィードバック → 改善
6. 本番運用

### やってはいけないこと
- ❌ Trading API（古い）でゼロから実装（時代遅れ・OAuth2 を学んで Fulfillment API へ）
- ❌ Monaco editor で手動 Ctrl+V 運用（clasp 導入が正解）
- ❌ スタッフ運用前にゼロベースで仕様策定（フィードバック必須）
- ❌ 重複チェック1層のみ（最低3層）
