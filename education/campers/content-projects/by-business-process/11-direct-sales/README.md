# 11. ダイレクト販売 — eBay外チャネル販売のAI化

> このフォルダは「eBay以外の直取り顧客対応」業務のAI化事例を集める棚です。
>
> ⚠️ eBayに直接関係しない派生事業ですが、Campersメンバーの直取相手対応にも応用できる素材として配置します。

---

## この業務段階で何ができるか（メンバー向け）

### 課題（メンバーが抱えがちな悩み）
- eBay経由のお客様が「直接買いたい」と言ってくれても受け皿がない
- 取りこぼしている直取需要がある（eBay手数料分を割引できる）
- 直取注文を Excel で管理して属人化・ミス発生
- 顧客リストを作っていない・LTV（顧客生涯価値）が見えない
- リピート購入の促進ができていない

### Claude Code でできること
- **直取販売サイトを Render / Vercel等の PaaS に簡単デプロイ**
- **顧客ログイン・注文管理・発送管理を一括できる Web アプリ作成**
- **eBay と独立した顧客 DB 管理**
- **発送ステータスの自動更新・通知**
- **リピート顧客への自動DM・割引提案**

## メンバーが最初に試すべきこと

### ステップ1：直取需要の有無を確認
- eBayで購入してくれた顧客から「直接取引したい」の打診はあるか？
- 知人・SNS経由で直取注文を受けているか？
- 取りこぼしている可能性のあるリピート顧客はいるか？

### ステップ2：小さく始める
- 最初は Google Form + スプレッドシートでも十分
- 注文受付 → 在庫確認 → 発送 のフローを手動で回す
- 月10件以上になったら本格的なシステム化を検討

### ステップ3：Webアプリ化
- Render等の PaaS（Platform as a Service）に簡単デプロイ
- Claude Code で React/Next.js アプリを作成
- 顧客ログイン・注文管理・発送管理を一括化

## 社長の体験：ダイレクト販売ツール（NY顧客向け）

### 何をやったか
- NY在住の常連顧客向けに直販サイトを構築
- Render に本番デプロイ
- 注文管理・発送管理・顧客対応を一元化

### 取り組みのプロセス
- ローカル開発 → Render デプロイ
- データ永続化（Render ディスク追加）
- 機能追加（発送管理・オーダー管理7項目）
- バグ修正5件・アーキテクチャ修正（max方式 → add方式）

### つまづいたこと
- データ構造理解不足で初期設計ミス
- Render ディスク追加直後にデータ消失（seed方式で復元）
- ログイン関数未定義バグで本番停止
- 営業時間中に本番デプロイして混乱

### 学んだこと
- **永続化は最初に設計すべき**（後から追加すると詰む）
- **ローカルテスト → 社長OK → 本番push の順序を厳守**
- **営業時間中の本番デプロイは禁止**
- **修正は本番データで即検証**（テスト環境ローカルで十分）

### 数値 Before/After
- 手動運用：注文1件あたり 5〜10分の Excel 管理
- Webアプリ化：注文1件あたり 30秒（顧客が自分で入力）

## メンバーへの含意

- **「eBayの裏でリピート顧客が育っているのに気付いていない」可能性**：常連を直取に切り替えると手数料分（10〜15%）がそのまま利益増
- **PaaS（Render等）で意外と簡単**：プログラミング未経験でも Claude Code でできる
- **小さく始めて運用回しながら改善**：完璧を目指すと出せない
