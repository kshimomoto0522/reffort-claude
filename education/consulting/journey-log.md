### 学び
💡 **eBayレポートの種類と使い分け**
eBay Seller Hubからダウンロードできるレポートは3種類：
- Traffic Report（CSV）：Impressions・CTR・Views・CVRなど表示系指標
- Advertising Sales Report（CSV）：PLG・PLPの広告効果
- Transaction Report（USD・CSV）：実際の収支・FVF・手数料など

この3つを統合して初めて「実態」が見える。バラバラに見ていると判断を誤る。

💡 **Transaction Reportの落とし穴**
`Transaction currency = USD`のみ使う。`ebaymag`（円建て）が混ざるとドル建て分析が狂う。

💡 **PLPとOffsiteはTransaction Reportに含まれない**
PLG（成果報酬型）はレポートに入るが、PLP・Offsiteは別途Seller Hubの広告管理画面で確認が必要。これを知らずに収支計算すると「利益が多く見える」状態になる。

💡 **「見られているのに売れない」と「そもそも見られていない」は別問題**
- Impressions低い → タイトル・カテゴリの問題（検索に引っかかっていない）
- Impressions高い・CTR低い → サムネイル・価格の問題
- CTR高い・CVR低い → 商品ページ内の問題（写真・説明・価格）
それぞれ打ち手が全く違う。

💡 **Python × Claudeのコンビで「コード不要」の自動化が実現**
Claudeがスクリプトを書き、ユーザーはコマンド1行を打つだけ。プログラミング未経験でも分析を自動化できる。

⚠️ **Scheduled Task（定期タスク）の使い方の学習コスト**
Chatworkへの週次自動投稿をScheduled Taskで設定したが、cronの書き方・タイムゾーン・実行環境の理解が必要。一度設定すれば完全自動になる価値は高い。

#### Step 1：Gitをインストール
Gitとは「ファイルのバージョン管理・複数PC同期ツール」。プログラミング用だが、ファイルのバックアップ・同期に使える。

1. `https://git-scm.com/download/win` にアクセス
2. 「Click here to download」でインストーラーをダウンロード
3. インストール（すべてデフォルト設定でOK）

#### Step 2：Claude Codeをインストール
1. `claude.ai/download` にアクセス
2. Windows版をダウンロード・インストール

#### Step 3：GitHubからreffortフォルダをダウンロード（クローン）
Claude Codeのターミナル（コードタブ）、またはコマンドプロンプトで以下を実行：

```bash
cd C:\Users\（新PCのユーザー名）\Desktop
git clone https://github.com/kshimomoto0522/reffort-claude.git reffort
```

「クローン」とは「GitHubのファイルをまるごとPCにコピーする」操作。これだけで最新のCLAUDE.mdやスクリプトがすべて手元に来る。

#### Step 4：Claude Codeでフォルダを開く
1. Claude Codeを起動
2. 左下のフォルダ表示をクリック → Desktopの`reffort`フォルダを選択
3. 自動的にCLAUDE.mdが読み込まれ、これまでの設定が復元される

#### Step 5：Pythonをインストール（スクリプトを動かす場合）
eBay週次レポートスクリプト（Pythonファイル）を動かすには別途必要。

1. `https://www.python.org/downloads/` にアクセス
2. 「Download Python」でインストール
3. インストール時に**「Add Python to PATH」にチェックを入れること**（重要）

💡 **タスクスケジューラの落とし穴：CWD（カレントディレクトリ）**

テスト完了→OK報告の間に、実際のデータ確認を挟まないと嘘の報告になる。確認手段があるのに使わないのは怠慢。

## 2026-03-26 ― Claude Codeセキュリティ対策

### やったこと

🔧 **git push --force / git reset --hard の禁止設定**
- settings.local.jsonのdenyリストに追加
- 万が一Claudeが誤って実行しても自動的にブロックされる
- 通常のgit push / git reset（soft/mixed）は影響なし

🔧 **設定ファイルからAPIトークン丸出しを削除（12行）**
- Chatworkトークン × 5行、Slackトークン × 5行、OpenAI APIキー表示 × 2行を削除
- .envでトークンを隠しているのに、settings.local.json（許可設定ファイル）にトークンが丸ごと記録されていた
- MCP経由の許可（トークンが見えない安全な経路）は残し、curlでの直接呼び出しを削除

🔧 **.envファイルの直接読み取りをブロック**
- denyリストに `Read(**/.env)` と `Read(**/.env.*)` を追加
- Pythonスクリプト経由の読み取り（週次レポート等）は影響なし
- Claudeが.envを直接開くことだけができなくなった

### 学び

💡 **セキュリティの本質：「誰の指示で読むか」が問題**
社長が「.envを読んで」と指示する分には問題ない。怖いのは「Webページに埋め込まれた隠れた指示でClaudeが.envを読まされ、外部に送信させられる」こと（プロンプトインジェクション）。社長自身がこの本質を見抜いた。
→ 対策として「Claudeが.envを直接読めない」設定にした。仮に騙されても読めないので安全。

💡 **「Always allow」の蓄積は見えないリスク**
Claude Codeは作業のたびに許可を求めてくる。面倒で「Always allow」を押し続けると、知らないうちに大量の許可が蓄積される。家の鍵を一つずつ外していくのと同じ。月1回の棚卸しが必要。

💡 **settings.local.jsonは「何を許可したかの履歴」でもある**
過去に一度でも許可したコマンドがそのまま記録される。curlでAPIを直接叩いた場合、トークンを含むコマンド全体が記録に残る。.envに移行しても設定ファイル側の古い記録を消さないと意味がない。

⚠️ **失敗：セキュリティ対策が後手だった**
3月11日にClaude Codeを使い始め、3月22日にAPIトークンの.env移行をしたが、settings.local.jsonにトークンが残っていることには気づかなかった。今日（3月26日）の確認で初めて発覚。セキュリティ設定は「最初の1日目」にやるべき。

- 社長のブラウザからは普通にASICS.comが開けていた → IPブロックではなかった
- 本当の原因はヘッドレスFirefoxのBot検出（Akamaiフィンガープリント）
- 15分ごとの自動チェックをセットしたが、ヘッドレスFirefoxで確認していたため毎回「BLOCKED」と報告

ブロックされているか確認するためにヘッドレスFirefoxでアクセス → これ自体がBot検出を追加発火させる。問題を確認する手段が問題を悪化させるパラドックス。

⚠️ **同じミスを2回繰り返した教訓**

### 学び
💡 **【最重要発見】スケジュールタスクはタスクごとに承認キャッシュを持つ**
- `settings.local.json` の allow list とは別に、**タスク単位**で承認を記憶する仕組みがある
- 「今すぐ実行」ボタンで一度実行したときに承認したツールが、次回以降の自動実行に引き継がれる
- これが社長の「毎回再起動するたびに承認を煽られる」問題の真因だった
- 新規タスク作成時は必ず「今すぐ実行」で一度走らせて承認を焼き込むのが運用上のベストプラクティス

💡 **チェックポイント先行更新の原則**
- `baychat-slack-hourly-check` が2日間停止した原因：処理を先に実行→最後にtsファイル更新する設計だったため、処理中のクラッシュでtsが未更新 → 次回も同じメッセージで詰まり続ける無限ループ
- 正しい設計：**処理の前にチェックポイント（tsファイル）を先行更新**
- 処理が途中で失敗しても次回は前進できる
- ts-basedの履歴追跡を伴う全ての定期タスクに適用できる普遍原則

💡 **Claude Code Routines判断の技術要件**
- 現状のタスクはほぼ全てローカル環境（Chrome拡張・ローカルPython・MCPサーバー・ローカルファイル）依存
- Routinesはクラウド実行のためローカルMCP不可
- 月の実行回数も無料枠（Max月15回）では大幅超過（chatwork-ai-reply月500回、baychat-slack-hourly-check月720回）
- **結論：ローカルscheduled-tasksで運用継続。Routinesは機会が来たら再評価**

💡 **静かすぎ問題 vs 通知過多問題**
- 最初の改修案：処理対象なしでも毎朝キュー状態レポート → 社長「毎日報告いらない」
- 正解：**実行確認はタスクUIの lastRunAt を見る。Chatwork通知は処理があった日とエラー時だけ**
- 情報密度とノイズ排除のバランスは、社長が何を「実行確認手段」と認識しているかで決まる

⚠️ **失敗・迷い**
- 当初「静かすぎ問題」として毎日通知を足す提案をしたが、社長から「毎日報告いらない、実行したことをただ知りたいだけ」と即フィードバック
- `lastRunAt`がUIで見える事実を先に提示できていれば、余計な改修を挟まずに済んだ
- 「社長が何を確認手段としているか」を先に聞くべきだった

### 🔧 仕組みの変更
- アクティブタスク6件体制（`monday-ebay-report-delivery` / `monday-report-requests-review` / `daily-x-digest` / `chatwork-ai-reply` / `baychat-slack-hourly-check` / `campers-member-removal-goto`）
