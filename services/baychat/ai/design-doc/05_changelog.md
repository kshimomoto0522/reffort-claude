# 05. 変更履歴

> このファイルの役割：**AI Replyに関わる全変更を時系列で記録する。**
> 「いつ・誰が・なぜ・何を変えた・どこに影響した」の5項目を必ずセットで残す。
> これによって、将来「なぜこの仕様なのか」を過去に遡って確認できる。

---

## 📝 記録フォーマット

変更を記録するときは、以下のテンプレートをコピーして先頭に追記する。新しい変更が上に来るようにする。

```markdown
---

### [YYYY-MM-DD] 変更タイトル（短く）

| 項目 | 内容 |
|------|------|
| 変更対象 | （どのファイル/ブロック/パラメータ） |
| 変更者 | （社長 / Claude / Cowatech / 複数） |
| 変更理由 | （なぜ変えたか） |
| 変更内容 | （何を、どう変えたか） |
| 影響範囲 | （どこに波及するか） |
| テスト | （どう検証したか。未検証の場合はその旨） |
| 関連ドキュメント | （関連するブロックカード・handoff等のリンク） |
| ロールバック手順 | （問題があった時の戻し方） |

**本文（詳細・背景・議論のメモ）:**
詳細な説明や議論の背景、意思決定のプロセスをここに書く。後から読んでも判断理由が分かるように。
```

---

**本文（詳細・背景・議論のメモ）:**

v0.1完成後、社長から以下の重要な指摘があった：

> 「スプレッドシートにバラバラにまとめているコード等の情報を設計図としてまとめて、わたし（あなた）とCowatechがしっかり情報共有する環境、Cowatechが情報更新する際にしっかり管理するための環境、あなたがAI Replyの仕様を確認するにはここを見ればすべてがわかるという状況と環境を作ることが目的です。」

これを受けて、以下の方針転換を実施：
- ダッシュボード作成（社長向けサマリー）は補助的な位置づけに
- 設計図本体を「CSV19枚の全情報を統合した単一参照源」として完成させる
- 5エージェントを並列投入して一気に情報収集
- 新規ファイル7つで不足領域をカバー

調査エージェント5本の成果：
1. CSV19枚完全棚卸し（約10,000文字の詳細報告）
2. 本番ペイロード完全解析（全パラメータ・全ブロック原文・レスポンス含む）
3. プロンプト全バージョン（v1.0〜v2.4）の差分と設計思想変遷

5. handoff・Slack草案から未解決論点・合意事項・テスト結果を集約

- 「単一参照源」→ 技術的には達成。運用は次フェーズ

- [ ] プレースホルダの追加・変更

## 📏 変更記録のルール

1. **必ず時系列で上に追加**（新しい変更が先頭）
2. **全項目を埋める**（該当なしなら「該当なし」と明記）
3. **ロールバック手順は具体的に**（「元に戻せない」場合はその旨を明記）
4. **関連ドキュメントはリンク化**（将来遡れるように）
5. **議論の背景は本文欄に残す**（なぜこの決定に至ったか）
