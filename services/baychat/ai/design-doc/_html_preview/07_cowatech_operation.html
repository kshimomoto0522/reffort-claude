07. Cowatech運用ルール

✅ v0.2 完成版
このファイルの役割：設計図の管理・更新・レビューをCowatechと共同で回すための運用ルールを定義する。この設計図が「単発の文書」ではなく「継続する管理基盤」になるためのプロセス。

🎯 運用の基本方針

項目 方針
管理主体 Reffort（社長＋Claude）が主。Cowatechがレビュー
更新プロセス 段階導入：初期は直編集、軌道に乗ったらPR運用
既存CSVの扱い 段階的デモート（3か月移行→廃止判断）
マスター言語 日本語 + 技術用語は英日併記
保管場所 reffort-claude GitHubリポジトリ（プライベート）
同期頻度 毎日深夜0時の自動バックアップ（既存運用に統合）

🔄 更新フローの3フェーズ

フェーズ1：初期運用（〜1か月）

スタイル：直編集
- Reffort側で直接設計図を編集
- Cowatechには「完成後」または「大きな変更時」にSlackで共有
- 軽微な修正（typo、表現改善）は共有不要

目的：設計図を早く成熟させる。形が固まるまでは高速に反復。

フェーズ2：安定運用（1〜3か月）

スタイル：PR＆レビュー
- 変更は GitHub PR として提出
- Cowatechがレビュー
- 合意後にマージ
- PR説明文に「何を変えたか」「なぜ変えたか」「影響範囲」を記載

目的：変更の質と透明性を担保。

フェーズ3：成熟運用（3か月〜）

スタイル：PR＆月次同期
- 通常の変更はPR＆レビュー
- 月1回、Slackで「設計図の定期レビューミーティング」を実施
- 既存論点の棚卸し・新規機能の方向性議論

目的：長期的な整合性を保つ。

📛 ファイル命名規則

既存ブロックカード

  • block_{順序}_{目的}.md
  • 例：block_00_item_info.md, block_n4_admin_prompt.md

新ブロック追加時

  • 順序番号をつける（ルールと整合）
  • 新バージョン追加時は既存カードを上書き（バージョン履歴セクションに追記）

番号付きファイル（設計図トップレベル）

  • {2桁番号}_{テーマ}.md
  • 新テーマ追加は次の番号を使用
  • 既存番号を入れ替えない（リンク破綻防止）

🔀 新機能追加時の手順（Cowatech向け）

Cowatechが新機能を実装する前に、以下の順で進める：

  1. Reffortが新機能の設計図ドラフトを作成
  2. 該当ブロックカード（新規 or 更新）
  3. 条件分岐表（04）に新分岐を追記
  4. UI注入マトリクス（02）に新UI項目を追記

  5. 変更履歴（05）に記録

  6. Cowatechにレビュー依頼（Slack #baychat-ai導入）

  7. 必要な情報はすべてGitHubに揃えた状態で依頼

  8. 技術的実現性・工数をフィードバック

  9. Cowatechの意見を反映してドラフト確定

  10. 合意事項をブロックカードの「Cowatech合意事項」欄に記録

  11. Cowatech実装開始

  12. 実装完了後、本番同等テストで検証

  13. services/baychat/ai/testing/batch_test.py で本番モードテスト

  14. スコア劣化がないか確認

  15. 本番反映

  16. 変更履歴に「本番反映日」を追記

🛡️ バージョン管理

設計図全体のバージョン番号

  • v0.1 — 叩き台（2026-04-20）
  • v0.2 — 実データ完全統合版（2026-04-20）
  • v1.0 — Cowatechレビュー完了版（未定）
  • 以降、意味的変更ごとに minor up

各ブロックカード・ファイルのバージョン

  • 全体バージョンに連動
  • 個別のバージョン履歴は各ファイル内「バージョン履歴」セクションに記録

⚠️ 運用上の注意点

やってはいけないこと

  • CSVとの二重管理：必ず設計図をマスターにし、CSVは「参考資料」とする
  • 未レビュー変更の本番反映：Cowatech実装前にReffort内で承認が必要
  • 変更履歴を残さない修正：05_changelog.md への記録は必須

気をつけること

  • リンク破綻：ファイル名変更時は他ドキュメントからのリンクも修正
  • 用語の揺らぎ：06_glossary.md を更新
  • 重複記述：同じ情報が複数ファイルに散らばらないように、一次情報の場所を明確に
