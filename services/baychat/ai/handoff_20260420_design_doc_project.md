## このドキュメントの目的

プロンプトv2.4の微修正ループに戻らない。**設計図（仕様書・ドキュメント基盤）を整備するという上位の方針転換**を、次セッションの最初から共有するための記録。

## 経緯と社長の問題意識（2026-04-20 原文より）

> 色んな問題がありゴールに向けて前段階で何の問題があり何を確認、改善していかないといけないのかがわからなくなってきた。
>
> 非エンジニアの認識として、**どうやってシェフ（AI）にわたしが求める味で調理してもらうためにマニュアル（プロンプト）を作るかを試行錯誤しているのに、そもそも調味料が足りていない、調理機器の温度を指定通りにしているのに機器の設定がズレている**などの状況に感じる。
>
> Cowatechの設計図にも非エンジニアながら質に疑問がある。**設計図として判断できるものではなく、スプレッドシートにざっくばらんに記載されているだけ**な気がする。
>
> 今後微修正など繰り返したり、要約モードを追加する際に**設計図の更新・管理という大事なことができない状況**に感じる。
>
> 急がば回れでまず**わたしたちとCowatechが完璧に意思疎通でき、管理できる設計図を作りたい**。
>
> コードを見てもわたしにはわからないが、例えば**左側にコードが順序よく仕様通り記載されている状態で、右側に今回議論しているようなこのコードがどこの機能に問題を与えているなどわかるような設計図**がほしい。
>
> 本来エンジニアがプロフェッショナルに設計図として管理、更にわたしの要望で**非エンジニアにも問題・改善点を見える化するにはどうすればいいか**、これを新しいセッションでスタートする。

**この直感は技術的にも完全に正しい。** 以下の4/14〜4/20の流れが、社長がこの結論に至った証拠。

→ **対策：設計図を再設計する。**

## 現状のCowatech設計図の問題点（定量化）

| 問題 | 具体例 |
|------|--------|
| **単一参照源がない** | SUMMARY_PROMT.csv, promt_spec_48665.csv, promt-update.csv, old_promt_spec.csv, promt_example.csv — プロンプト仕様が5枚以上のCSVに分散 |
| **コード↔機能↔影響範囲のマッピングがない** | FORCED TEMPLATEブロックを外した場合、BayChat側のどの処理に影響するかがドキュメントから読み取れない |
| **条件分岐が暗黙** | descriptionが空のとき「Create questions/answers...」ブロックは挿入されない、という仕様がどこにも明記されていない（本番ペイロードから推測するしかない） |
| **UI連動の明記がない** | TO/FROMプルダウン、トーン選択、補足情報欄がプロンプトのどこに差し込まれるかの対応表がない |
| **変更履歴がない** | promt-update.csvはあるが、いつ・なぜ・誰が変えたかの記録フォーマットではない |
| **バリエーション整理不足** | tone別（polite/friendly/apologetic）の差分がSUMMARY_PROMT.csvの右3列に並んでいるだけで、差分の意図が書かれていない |
| **後処理が不明** | AIが返す`buyerLanguage`をBayChat側で切り出し・加工しているかが仕様に書かれていない（今回Q1で質問した内容） |

## ゴール

**社長・Claude・Cowatechの3者が完璧に意思疎通でき、継続的に管理できる設計図を作る。**

### 評価基準（これができれば成功）
1. 社長が週に1回眺めるだけで「今どこにどんな仕様があるか」が把握できる
2. Claudeが設計図を読めばCowatech質問なしで仕様判断ができる
3. 新機能（要約モード等）追加時に、既存仕様のどこに影響するかが事前に見える
4. プロンプト変更時に、変更理由・影響範囲・テスト結果がセットで記録される
5. Cowatechが実装時に「ドキュメント通り作れば正しく動く」と言える粒度

## 理想のイメージ（社長の要望を技術要件に翻訳）

### 社長原文
> 左側にコードが順序よく仕様通り記載されている状態で、右側に今回議論しているようなこのコードがどこの機能に問題を与えているなどわかるような設計図

### 技術要件への翻訳（Claudeが次セッションで提案する案の叩き台）

**「二段テーブル型リビングドキュメント」（仮称）**

| 左列：実装 | 中央：目的 | 右列：UI/機能/影響範囲 |
|---------|---------|-------------------|
| プロンプトブロック[0] 商品情報JSON | AIに商品コンテキストを与える | 商品データベース → ItemID経由で取得 |
| プロンプトブロック[N+2] BASE_PROMPT | eBayコンプライアンス制約 | 全リプライで不変。変更はコンプラ観点のみ |
| プロンプトブロック[N+3] OUTPUT_FORMAT | JSON出力強制 | BayChat画面の「日本語訳/英語返信」2タブ表示に直結 |
| プロンプトブロック[N+4] admin prompt | CS品質・トーン・会話ロジック | 管理画面で編集可能。最も頻繁に改善する層 |
| プロンプトブロック[N+5] FORCED TEMPLATE | 挨拶・署名の強制 | **TO/FROMプルダウンの値の注入先**。除去すると注入先を別途設計必要 |

…これが各ブロックごとに詳細カードで展開される。

## 次セッション開始時に議論すべき論点（優先順）

### A. 設計図のフォーマットを決める
- **A1**: Markdown + 表形式（gitで履歴管理・git管理下）← Claude最推奨
- **A2**: 機能カード型（1ブロック1ページ、カード間リンク）
- **A3**: Mermaid図（全体フロー） + Markdown（詳細）
- **A4**: 上記ハイブリッド：全体図Mermaid + 各ブロックのカード + UI連動マトリクス表

### B. ツール選定
- **B1**: Markdown + GitHub（reffort-claudeリポジトリ延長・社長スマホから参照可能）
- **B2**: Notion/Confluence等（社長閲覧が楽・但しgit管理不可）
- **B3**: ハイブリッド（マスターはmd、Notionに自動ミラーリング）

### C. Cowatechとの役割分担と合意形成
- **C1**: 管理主体（Cowatech主 / Reffort主 / 共同）
- **C2**: 更新プロセス（PR＆レビュー / 直編集 / 月次同期ミーティング）
- **C3**: 既存19CSVの扱い（廃止 / 段階移行 / 並存）
- **C4**: 言語（日本語マスター / 英語マスター / 両対応）

### D. 見える化すべき必須要素（網羅チェック）
- [ ] プロンプトブロック全順序（[0]〜[N+5]）
- [ ] 各ブロックのON/OFF条件（descriptionが空なら抜ける等）
- [ ] UIの操作項目 → プロンプトへの注入経路マッピング（TO/FROM/tone/description）
- [ ] tone別バリエーション差分（polite/friendly/apologetic）
- [ ] AI出力JSON schema（multi_language_reply）
- [ ] BayChat後処理（buyerLanguageをどう扱うか）
- [ ] DB構造との関連（どのテーブル・カラムが何に使われるか）
- [ ] APIパラメータ（temperature, model, response_format）
- [ ] エラー処理・リトライ挙動
- [ ] 変更履歴フォーマット（いつ・誰が・なぜ・影響範囲）

### E. 移行計画（決まってから段階実行）
- **E1**: 既存資産（19CSV + gpt_api_payload.txt + admin画面設定）を棚卸し
- **E2**: ドラフトv0.1を作ってClaudeが起案
- **E3**: 社長レビュー → 方針確定
- **E4**: Cowatechに共有・レビュー依頼
- **E5**: 合意後マスターとして運用開始。既存CSVは「参考資料」にデモート
